## 一、数据库设计总览

### 知识点 1：数据库设计阶段
讲解：数据库设计一般分为需求阶段、概念设计阶段、实现阶段。ER 模型主要用于概念设计阶段，把业务世界转成结构化模型。

例子：学校系统先访谈教务老师得到“学生选课、教师授课、课程开课”等需求，再画 E-R 图。

注意事项：
- 概念设计不要一上来写表结构。
- 先统一业务术语，再建模。

### 知识点 2：两类常见设计问题
讲解：坏设计常见两类问题是冗余和不完整。冗余会导致更新异常，不完整会导致业务难以表达。

例子：把教师电话重复存到每条授课记录中会形成冗余；不建“先修课”关系就无法表达课程依赖。

注意事项：
- 冗余和查询方便是权衡关系，不是绝对越少越好。
- 概念层先保证语义正确，再考虑实现优化。

## 二、ER 模型基础概念

### 知识点 3：实体集
讲解：实体集是同类对象的集合，通常对应现实中“可独立识别”的对象。

例子：`Student`、`Instructor`、`Course` 都是实体集。

注意事项：
- 能独立存在、需要单独管理的信息，优先建实体。

### 知识点 4：联系集
讲解：联系集描述实体之间的关联，二元联系最常见，也可以有三元联系。

例子：`Advisor(Student, Instructor)` 表示导师关系。

注意事项：
- 联系本身也可以有属性，例如“指导日期”。

### 知识点 5：角色
讲解：同一实体集在一个联系中可能出现多次，此时要用角色区分语义。

例子：`Parent(adult, child)` 中 `adult` 和 `child` 是不同角色。

注意事项：
- 角色名必须有业务语义，避免使用模糊命名。

### 知识点 6：属性类型
讲解：属性可分为简单/复合、单值/多值、可推导属性。每类属性在转关系模式时处理方式不同。

例子：姓名可拆为“姓+名”（复合属性）；手机号可能是多值属性；年龄可由生日推导。

注意事项：
- 可推导属性通常不落库，避免冗余。
- 多值属性往往需要拆分成独立关系。

## 三、约束表达

### 知识点 7：映射基数
讲解：二元联系常见基数有 1:1、1:N、N:1、M:N，用于约束“最多关联多少个”。

例子：`Department` 与 `Instructor` 常是 1:N。

注意事项：
- 基数回答“最多多少”；参与约束回答“至少多少”，两者都要建模。

### 知识点 8：全参与与部分参与
讲解：全参与表示实体集中每个实体必须参加该联系；部分参与表示可选。

例子：学生“必须有导师”是全参与；教师“可以没有学生”是部分参与。

注意事项：
- 全参与通常在落地时会体现为非空外键或额外约束。

### 知识点 9：三元联系中的约束
讲解：三元联系的箭头和基数语义比二元更复杂，不能机械套用二元思路。

例子：`Proj_Guide(Instructor, Student, Project)` 可表达“同一教师在同一项目中最多指导一个学生”等复杂规则。

注意事项：
- 三元联系常被误画成多个二元联系，语义可能丢失。

## 四、弱实体与主键设计

### 知识点 10：弱实体集
讲解：弱实体不能仅靠自身属性唯一标识，存在依赖于强实体，需要识别联系。

例子：`Section` 依赖 `Course`，`sec_id` 只在同一课程内唯一。

注意事项：
- 弱实体必须有判别符和识别实体。

### 知识点 11：主键选择规则
讲解：主键设计需要结合实体/联系类型。强实体用自身键，弱实体用“识别实体键 + 判别符”。

例子：
- 强实体：`Course(course_id)`  
- 弱实体：`Section(course_id, sec_id, semester, year)` 主键可为 `(course_id, sec_id, semester, year)`

注意事项：
- 多对多联系转表后主键通常是两侧主键组合。

## 五、高级建模结构

### 知识点 12：特化与泛化
讲解：下层实体继承上层实体的属性和联系。常见维度有互斥/重叠、完全/部分。

例子：`Person` 可特化为 `Student` 和 `Instructor`。

注意事项：
- 先明确是否允许同一对象属于多个子类（重叠）。
- 再明确是否所有父类对象都必须进入某个子类（完全）。

### 知识点 13：聚合
讲解：聚合用于把“联系本身”当成更高层对象，参与新的联系，表达“联系上的联系”。

例子：把 `Proj_Guide(Student, Instructor, Project)` 作为整体，再与 `Evaluation` 建联系。

注意事项：
- 若只是增加普通属性，不一定需要聚合。
- 聚合主要用于语义清晰和复杂关系表达。

## 六、ER 图到关系模式的转换

### 知识点 14：实体集转表
讲解：每个实体集通常转成一张关系表，实体键转主键。

```sql
CREATE TABLE course (
  course_id VARCHAR(10) PRIMARY KEY,
  title VARCHAR(100),
  credits INT
);
```

注意事项：
- 实体属性尽量原子化，便于查询和约束。

### 知识点 15：复合属性与多值属性转换
讲解：复合属性拆列；多值属性拆成新表并带回指外键。

```sql
CREATE TABLE inst_phone (
  id VARCHAR(10),
  phone_number VARCHAR(20),
  PRIMARY KEY (id, phone_number)
);
```

注意事项：
- 多值属性不要直接放在一个字符串字段里拼接。

### 知识点 16：多对多联系转表
讲解：多对多联系要单独建表，包含双方主键和联系属性。

```sql
CREATE TABLE takes (
  student_id VARCHAR(10),
  course_id VARCHAR(10),
  sec_id VARCHAR(10),
  semester VARCHAR(10),
  year INT,
  grade VARCHAR(2),
  PRIMARY KEY (student_id, course_id, sec_id, semester, year)
);
```

注意事项：
- 联系属性（如成绩）应放在联系表，而不是任一实体表。

### 知识点 17：多对一联系转换
讲解：多对一联系通常可在“多的一侧”加外键，也可以单独建联系表。

```sql
ALTER TABLE student
ADD advisor_id VARCHAR(10);
```

注意事项：
- 如果联系本身有属性，建议保留独立联系表。

### 知识点 18：特化映射常见方法
讲解：常见做法有“父表+子表”或“单表+类型字段”。课件提示了 Method 1/Method 2 的思想。

例子：父表 `person` 存公共字段，子表 `student`、`instructor` 存特有字段并以同一主键关联。

注意事项：
- 继承层次深时，优先选择可维护性高的方案。
- 方案选择与查询模式、更新频率密切相关。

## 七、建模决策与易错点

### 知识点 19：实体还是属性
讲解：如果信息有多值、附加信息或独立生命周期，通常应建实体而不是简单属性。

例子：电话号码若只存一个值可做属性；若要记录类型、归属地、是否主号，宜建实体。

注意事项：
- 课件强调“多个电话号码”“电话号码附加信息”是典型决策点。

### 知识点 20：实体还是联系
讲解：若描述的是“动作/事件”，更适合建联系或事件实体，而不是塞进某个实体属性。

例子：学生选课应建 `takes` 联系，而不是在 `student` 表里加“已选课程字符串”。

注意事项：
- 事件型数据通常包含时间、状态、金额等属性，适合独立建模。

### 知识点 21：二元还是三元
讲解：能用二元准确表达就不用三元；若二元会损失语义，应保留三元。

例子：父母与孩子关系可用两个二元关系；项目指导关系常需三元关系表达完整约束。

注意事项：
- “能转成多个二元”不代表“语义等价”。

### 知识点 22：非二元转二元方法
讲解：可引入中间实体，把一个 n 元联系拆成若干二元联系。

例子：将 `R(A,B,C)` 转成新实体 `E`，再建 `E-A`、`E-B`、`E-C` 三个二元联系。

注意事项：
- 转换后要把原联系约束补齐，否则可能产生伪组合。

### 知识点 23：符号体系与记法一致性
讲解：Chen、Crow’s Foot、UML 都能表达建模思想，但一张图应保持同一种记法。

例子：同一课程作业中全程使用 Crow’s Foot，不混用 Chen 菱形与 UML 语法。

注意事项：
- 课件明确提醒：不要混用不同记法。
