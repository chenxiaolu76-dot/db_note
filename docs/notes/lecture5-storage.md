# 模块四：数据库记录与文件组织（PPT 36-49）
本模块是数据库物理存储的核心基础，解决的核心问题：**数据库的逻辑数据（表、记录、字段），如何映射到磁盘的物理块（页）上**。上层的查询、事务、索引，完全依赖这个底层结构。

## 知识点 1：数据库文件的基础存储模型（PPT37）
这是数据库存储的最底层抽象，标准化层级定义：
- **顶层映射**：数据库 → 文件集合（一个库对应多个物理文件）
- **中层映射**：文件 → 记录序列（一张表对应一个/多个文件）
- **底层映射**：记录 → 字段序列（一行数据）

**核心前提**：记录大小 < 磁盘块（页）大小
- 原因：磁盘最小读写单位是块，跨块会导致两次 I/O，严重降低性能。

## 知识点 2：定长记录的存储与删除处理（PPT38-41）
定长记录：所有字段固定长度（int、char），无变长字段。

### （1）基础存储规则
- 寻址公式：第 `i` 条记录起始地址 = `n * (i-1)`，`n` 为记录固定长度
- 优化：**禁止记录跨块**，块末尾空间留作碎片，保证一次 I/O 读完整记录

### （2）定长记录的 3 种删除方案
1. **移动后续记录填补**
   - 操作：删除后整体前移
   - 优点：无碎片、连续存储
   - 缺点：I/O 巨大、性能极差
   - 场景：极小静态表，几乎不用

2. **末尾记录填补空位**
   - 操作：把最后一条记录移到删除位置
   - 优点：I/O 少、实现简单
   - 缺点：物理顺序打乱、无法直接偏移寻址
   - 场景：简单单表

3. **空闲链表法（工业主流）**
   - 操作：不移动数据，用指针链串起空闲位置，文件头存链表头
   - 优点：增删仅改指针、无数据移动、I/O 极小、性能最高
   - 缺点：需额外指针空间
   - 场景：PostgreSQL、MySQL MyISAM 默认方案

## 知识点 3：变长记录的存储实现（PPT42）
变长记录场景：varchar、text、blob、多类型记录、重复字段。

**标准存储格式：定长头部 + 变长数据区**
- **定长头部**（前置）
  - 空值位图（Null bitmap）：1bit 标记一个字段是否为 NULL
  - 变长字段偏移量+长度数组：快速定位字段位置
- **变长数据区**（后置）：所有变长字段实际数据

**优势**
- 先读头部即可知道所有字段位置
- 支持按需加载，跳过无用字段，减少 I/O
- NULL 仅占 1bit，空间高效

## 知识点 4：槽式页（Slotted Page）结构（PPT43）
**主流数据库通用核心结构**：MySQL InnoDB、PostgreSQL、SQL Server 都使用。

### 设计背景
变长记录增删改会移动位置，直接用物理地址寻址会全部失效，槽式页实现**逻辑寻址与物理存储解耦**。

### 核心结构
一个磁盘页分为：
- **页头**
  - 记录条目数（槽数量）
  - 空闲空间结束位置
  - **槽数组**：每个槽存「记录偏移量 + 长度」，从头部向后增长
- **记录数据区**：从页尾**反向向前**存储
- **空闲空间**：槽数组与记录区中间

### 核心规则与优势
- **寻址标准**：外部只用 **页号 + 槽号（RID）**，不直接用物理地址
- **记录可自由移动**：只需改槽内偏移，外部指针不变
- **完美支持变长记录**，空间利用率极高

## 知识点 5：大对象（LOB）存储方案（PPT44）
针对超过页大小的超大字段（blob、clob）。

### 3 种基础方案
1. 存文件系统，数据库存路径
2. 存数据库专属文件
3. 拆分成固定大小分片，用分片表管理

### PostgreSQL TOAST 机制（工业标杆）
- 全称：The Oversized-Attribute Storage Technique
- 实现：
  - 普通表自动对应 TOAST 表
  - 字段超过阈值（默认 2KB）自动分片存入 TOAST 表
  - 主表只存指针，**按需加载**
- 优势：主表更小、一页存更多记录、查询不加载大对象，大幅减少 I/O

## 知识点 6：记录在文件中的 5 种组织方式（PPT45-49）
解决「一张表的所有页/记录如何组织成物理文件」。

### 1. 堆文件组织（Heap File）
- 规则：记录可放在任意空闲位置，无序
- 配套：两级空闲空间映射表，快速找空闲页
- 优点：插入极快
- 缺点：无索引全表扫描慢
- 场景：OLTP 默认表结构，需配合索引

### 2. 顺序文件组织
- 规则：按搜索键**顺序连续存储**
- 处理：删除用指针、插入用溢出块、定期重组
- 优点：顺序/范围查询极快
- 缺点：增删成本高
- 场景：静态表、批量扫描表

### 3. 多表聚簇文件组织
- 规则：多张频繁关联表存在**同一块/同一文件**
- 优势：JOIN 一次 I/O 拿全数据
- 缺点：单表查询会读无关数据
- 场景：数据仓库、频繁 JOIN 表组

### 4. B+ 树文件组织
- 规则：记录存在 B+ 树叶子节点，自带有序
- 优点：增删查改稳定 O(log n)，支持高效范围查询
- 场景：**MySQL InnoDB 索引组织表**

### 5. 哈希文件组织
- 规则：搜索键哈希值 → 直接定位磁盘块
- 优点：等值查询极快（一次 I/O）
- 缺点：不支持范围查询
- 场景：仅等值查询表（用户ID、会话ID）

# 模块五：数据存储布局优化（PPT 50-54）
在基础存储之上，**按业务场景做架构优化**。

## 知识点 1：表分区技术（PPT50）
- 定义：大逻辑表 → 多个物理独立分区，对应用透明
- 优势：
  - **分区裁剪**：查询只扫目标分区，大幅减少 I/O
  - **删历史极快**：直接删分区，无日志开销
  - **冷热分离**：热数据 SSD，冷数据 HDD
- 场景：超大表、时间序列表、数据仓库

## 知识点 2：行式存储 vs 列式存储（PPT51-53）
### 行式存储
- 规则：一条记录所有字段**连续存储**
- 优点：整行增删改快、支持事务 ACID
- 场景：**OLTP 事务系统**

### 列式存储
- 规则：每一列**单独存储**
- 优势：
  - 只读取需要的列，**减少 90%+ 无效 I/O**
  - 压缩比极高、CPU 缓存命中率高
  - 支持 SIMD 向量计算，聚合极快
- 缺点：整行查询重构成本高、增删改慢
- 场景：**OLAP、数据仓库、大数据分析**
- 主流格式：Apache Parquet、ORC

## 知识点 3：数据字典（系统目录）存储（PPT54）
- 定义：数据库的**元数据库**，存储描述数据的数据
- 内容：
  - 表/字段/约束/视图
  - 用户权限
  - 统计信息（优化器用）
  - 物理存储位置、索引信息
- 特点：本身也是系统表，可用 SQL 查询（information_schema）
- 作用：数据库运行的基础，SQL 解析、权限校验、查询执行都依赖它

# 模块六：缓冲区管理（PPT 55-67）
数据库**性能心脏**，所有数据访问必经缓冲池，目标：**最小化磁盘 I/O**。

## 知识点 1：核心定位与概念（PPT55）
- **磁盘块**：数据库最小读写单元
- **缓冲区**：内存中存放一个磁盘块的空间
- **缓冲池**：缓冲区集合，数据库核心内存区域
- **缓冲区管理器**：管理缓冲池，所有磁盘访问必须经过它

## 知识点 2：缓冲区管理器完整工作流程（PPT56）
1. 接收磁盘块访问请求
2. 查页表：块是否在缓冲池
   - 命中：直接返回内存地址
   - 未命中：分配缓冲区
3. 缓冲池满 → 执行替换策略，驱逐“牺牲块”
   - 脏块必须先写回磁盘
4. 从磁盘加载块到缓冲区
5. 返回内存地址给上层

## 知识点 3：缓冲池并发控制（PPT57）
### （1）Pin / Unpin 机制
- Pin：访问前钉住页面，计数 +1
- Unpin：访问完成，计数 -1
- 规则：**只有 Pin=0 的块才能被驱逐**
- 目的：防止正在使用的页被换出导致崩溃

### （2）缓冲块锁机制
- **共享锁（读）**：多线程可同时持锁
- **排他锁（写）**：独占、互斥
- 目的：保证并发读写一致性

## 知识点 4：缓冲区替换策略（PPT58）
### LRU（最近最少使用）
- 驱逐最久未访问块
- 通用高效，但**全表扫描会造成缓存污染**

### Toss-immediate（立即丢弃）
- 顺序扫描块处理完立刻释放
- 彻底避免缓存污染

### MRU（最近最多使用）
- 驱逐刚访问的块
- 适合嵌套循环 JOIN 等循环访问场景

### 混合策略
- 查询优化器给出 Hint，动态选择策略
- 现代数据库主流方案

## 知识点 5：数据库完整数据访问流程（PPT59-67）
1. 索引 → 拿到 RID（页号+槽号）
2. 向缓冲区管理器请求页
3. 缓冲池命中/加载磁盘页
4. 通过槽号定位记录偏移 → 读取数据
5. 修改 → 标记脏页 → 后台写回磁盘
6. 缓冲池满 → 按策略驱逐 Pin=0 的块

# 模块七：内存数据库存储优化（PPT 68-70）
当数据全在内存时，**去掉磁盘架构的所有累赘**，实现性能飞跃。

## 知识点 1：内存数据库存储设计（PPT68）
- 背景：全量数据常驻内存，无磁盘 I/O 瓶颈
- 设计：
  - 无需槽式页复杂间接寻址
  - 索引直接存**内存指针**，而非 RID
  - 定长/变长数据分离内存池，减少碎片
- 效果：性能提升 **1~2 个数量级**

## 知识点 2：消除页间接寻址（PPT69）
### 磁盘数据库（两层跳转）
索引 → RID → 页表 → 内存页 → 槽 → 记录

### 内存数据库（一步直达）
索引 → **内存指针** → 记录
- 完全消除两层间接开销，访问速度极致提升

## 知识点 3：规避页锁存（Latch）开销（PPT70）
- 磁盘数据库：页换入换出必须加页锁，高并发 CPU 开销巨大
- 内存数据库：
  - 无缓冲池、无页换入换出
  - **无需页级锁**
  - 只用行锁/无锁结构
- 优势：高并发场景 CPU 开销大幅降低
