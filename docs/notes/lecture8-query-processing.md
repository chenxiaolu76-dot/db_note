# 数据库课程第 8 讲 Query Processing 详细笔记

## 文档说明

- 对应课件：`Database2026 - lecture 8 - Query Processing.pptx`
- 主题：查询处理（Query Processing）
- 目标：理解数据库如何把一条 SQL 变成可执行计划，以及选择、排序、连接等核心操作的典型执行算法与代价分析

## 整体逻辑概述

这一讲解决的问题是：**一条 SQL 到底怎样被数据库真正执行出来**。

核心链路可以概括为：

1. SQL 先被解析、绑定、重写和优化，形成查询计划（query plan）
2. 查询计划由多个操作符（operator）组成
3. 执行引擎（execution engine）按照某种处理模型（processing model）驱动这些操作符
4. 每个操作符再选择具体算法，比如顺序扫描、索引扫描、外部排序、哈希连接等
5. 优化器（optimizer）最终根据代价（cost）挑选一个更合适的执行方案

所以这讲不是单纯背算法，而是要建立一个整体意识：
**SQL 执行 = 执行模型 + 物理算法 + 代价估计 + 系统调度**

## 一、查询处理的总体架构

### 1. 系统模块的分工

课件给出的架构里，数据库大致包含这些部分：

- SQL 解析器（SQL Parser）：把 SQL 语句转成语法结构
- 绑定器（Binder）：把表名、列名和实际对象绑定起来
- 重写器（Rewriter）：做规则层面的等价变换
- 优化器（Optimizer）：比较不同执行计划的代价
- 代价模型（Cost Model）：估算不同计划的大致开销
- 执行引擎（Execution Engine）：真正执行操作符
- 存储管理器（Storage Manager）：负责页、文件、索引、缓冲等底层数据访问

可以把它理解成两层：

- 上层负责“想清楚怎么做”
- 下层负责“真的把它做出来”

### 2. 查询计划的基本单位

#### 查询计划（Query Plan）

- 一个查询计划由多个操作符（operators）组成
- 例如：扫描（scan）、选择（selection）、投影（projection）、连接（join）、排序（sort）

#### 操作符实例（Operator Instance）

- 操作符实例是某个操作符在某段数据上的一次具体调用
- 同一个逻辑操作符，在并行执行时可能对应多个实例

#### 任务（Task）

- 任务是一个或多个操作符实例组成的执行单元
- 它是系统调度（scheduling）的基本对象之一

#### 工作线程/工作者（Worker）

- worker 是代表客户端执行任务并返回结果的数据库内部执行单元
- 它可能对应进程（process）或线程（thread）

## 二、进程模型、调度与执行优化

### 1. 数据库常见进程模型（Process Models）

课件给了三类常见做法：

#### 模式 1：每个 DBMS worker 一个进程（Process per DBMS Worker）

- 每来一个请求，分配一个独立进程处理
- 优点：进程隔离强，出错影响相对局部
- 缺点：上下文切换（context switch）开销较大

#### 模式 2：进程池（Process Pool）

- 预先准备一组进程，查询来了后复用
- 优点：避免频繁创建/销毁进程
- 缺点：管理更复杂

#### 模式 3：每个 DBMS worker 一个线程（Thread per DBMS Worker）

- 用线程替代进程处理并发查询
- 优点：上下文切换成本更低
- 缺点：要处理共享内存（shared memory）和线程安全问题

### 2. 调度（Scheduling）在决定什么

对每个查询计划，DBMS 都要决定：

- 在哪里执行（where）
- 何时执行（when）
- 如何执行（how）

更具体地说，包括：

- 用多少个任务（tasks）
- 用多少个 CPU 核心（CPU cores）
- 每个任务绑定到哪个核心
- 中间结果输出到哪里

课件强调：**DBMS 往往比操作系统更懂查询本身的执行特征**。  
原因是操作系统只看到通用进程/线程，而 DBMS 知道哪个算子是阻塞的、哪个阶段会爆内存、哪个连接更适合并行。

### 3. 执行优化（Execution Optimization）的三条主线

#### 减少指令数（Reduce Instruction Count）

- 用更少的 CPU 指令完成同样工作
- 例如减少重复判断、减少函数层层调用

#### 降低每条指令的周期数（Reduce Cycles per Instruction）

- 让 CPU 更高效执行
- 典型思路包括改善缓存命中、降低分支预测失败、减少随机访存

#### 并行化执行（Parallelize Execution）

- 让多个线程同时处理一个查询
- 尤其适合 OLAP（Online Analytical Processing，联机分析处理）

## 三、三种查询处理模型

处理模型（processing model）定义了：**操作符之间怎样传递数据、一次传多少数据、谁驱动谁工作**。

课件重点讲了三种模型。

### 1. 迭代器模型（Iterator Model）

也叫：

- Volcano Model
- Pipeline Model

#### 基本思想

- 每个操作符实现一个 `next()` 函数
- 每次调用只返回一条元组（tuple）
- 如果没有更多结果，就返回空标记
- 父操作符通过不断调用子操作符的 `next()` 拉取数据

#### 直观理解

这是一种“自顶向下拉数据”的方式：

- 投影要结果，就向连接要一条
- 连接要一条，就向左右孩子继续要
- 直到叶子扫描算子返回真实元组

#### 优点

- 非常通用，几乎所有 DBMS 都支持
- 很容易形成流水线（tuple pipelining）
- 输出控制简单，适合边算边返回结果

#### 缺点

- 函数调用次数非常多
- 一次只处理一条元组，CPU 利用率不够高
- 对现代 CPU 的缓存和 SIMD（Single Instruction Multiple Data，单指令多数据）不够友好

#### 哪些操作会阻塞

虽然叫流水线，但不是所有算子都能完全边来边吐：

- 排序（sorting）
- 某些连接（joins）
- 子查询（subqueries）
- `ORDER BY`

这些算子常常要先收齐一批输入，才能继续向上输出。

### 2. 物化模型（Materialization Model）

#### 基本思想

- 每个操作符一次性处理完所有输入
- 然后一次性输出整个结果
- 即把中间结果“物化”（materialize）出来

#### 特点

- 输出可以是一整行（row）
- 也可以只是若干列（columns）
- 既可适用于 NSM（N-ary Storage Model，行存），也可适用于 DSM（Decomposition Storage Model，列存）

#### 优点

- 协调开销低
- 函数调用次数更少
- 对只访问少量元组的小查询更友好

#### 缺点

- 中间结果可能很大
- 需要额外内存或磁盘空间保存结果
- 对大规模分析型查询不一定好

#### 适用场景

- 更适合 OLTP（Online Transaction Processing，联机事务处理）
- 因为 OLTP 通常一次只碰少量记录，更在意低延迟和低调度开销

### 3. 向量化/批处理模型（Vectorized / Batch Model）

#### 基本思想

- 仍然保留 `next()` 风格
- 但每次返回的不是一条元组，而是一批元组（batch）

#### 与迭代器模型的区别

- 迭代器模型：一次一条
- 向量化模型：一次一批

#### 优点

- 显著减少操作符调用次数
- 更适合利用 SIMD 指令
- 更适合现代 CPU 的 cache line 和流水线

#### 缺点

- 实现复杂度更高
- 批大小（batch size）选得不好，可能浪费内存或降低吞吐

#### 适用场景

- 非常适合 OLAP
- 因为 OLAP 查询通常要扫很多数据、做聚合和连接，批处理更能发挥 CPU 吞吐能力

### 4. 三种模型如何比较

#### 迭代器模型

- 通用性强
- 实现经典
- 但函数调用开销大

#### 物化模型

- 小结果、小查询时很直接
- 但中间结果大时容易浪费空间

#### 向量化模型

- 更接近现代高性能数据库的方向
- 特别适合列存和分析型负载

## 四、计划处理方向与代价估计

### 1. 计划处理方向（Plan Processing Direction）

课件给了两个方向。

#### 自顶向下（Top-to-Bottom）

- 从根节点开始
- 父节点主动向子节点“拉”（pull）数据
- 这是迭代器模型最典型的工作方式

#### 自底向上（Bottom-to-Top）

- 从叶子节点开始
- 子节点把结果向父节点“推”（push）上去

#### 对比理解

- `pull` 模型控制逻辑更自然
- `push` 模型更利于紧凑流水线，有时更利于缓存和寄存器控制

### 2. 查询代价（Query Cost）通常看什么

这一讲主要从磁盘 I/O 角度估计成本。

磁盘代价通常看：

- 寻道次数（number of seeks）
- 读块数（number of blocks read）
- 写块数（number of blocks written）

简化后，常用两个核心量：

- 块传输次数（block transfers）
- 寻道次数（seeks）

### 3. 为什么代价估计只是估计

现实系统中，代价并不完全固定，因为：

- 所需数据可能已在缓冲区（buffer）里
- 可用内存会受并发查询和操作系统影响
- 算法可能用更多缓冲区减少 I/O

所以数据库会在两类估计之间折中：

- 悲观估计（worst case）：假设数据不在 buffer 中
- 乐观估计（optimistic estimate）：利用更贴近现实的缓存命中假设

## 五、选择（Selection）与访问方法（Access Methods）

### 1. 什么是访问方法（Access Method）

访问方法就是：**数据库如何真正从表里把数据拿出来**。

课件列了三类基础方法：

- 顺序扫描（Sequential Scan / File Scan）
- 索引扫描（Index Scan）
- 多索引扫描（Multi-Index Scan），PostgreSQL 中常叫位图扫描（Bitmap Scan）

### 2. 顺序扫描（Sequential Scan）

#### 基本做法

- 按页（page）遍历整张表
- 每页再逐条检查元组是否满足条件
- DBMS 内部用游标（cursor）记住已经扫到的位置

#### 优点

- 实现最简单
- 对任何条件都适用
- 没有索引时它是保底方案

#### 缺点

- 对选择性高（selective）的查询通常很差
- 明明只要几条记录，也可能把整张表扫完

### 3. 顺序扫描为什么常常是“最差但必备”

课件直接说：顺序扫描几乎总是 DBMS 最不想做但又必须能做的事情。

原因是：

- 它不利用索引
- I/O 量随表规模线性上升
- 条件越精确，浪费越明显

### 4. 顺序扫描的常见优化

#### 预取（Prefetching）

- 提前把后续页面读入内存
- 降低等待磁盘的时间

#### 绕过缓冲池（Buffer Pool Bypass）

- 顺序大扫描时，不必污染缓冲池
- 避免把热点页挤出去

#### 并行化（Parallelization）

- 多个线程分段扫描同一张表

#### 堆聚簇（Heap Clustering）

- 让物理存储顺序更贴近访问模式

#### 区间摘要（Zone Maps）

- 每页维护某些列的预计算摘要（如最小值/最大值）
- 先查 zone map，再决定页要不要读

#### 晚物化（Late Materialization）

- 先只处理标识符或少量列
- 必要时再回表取完整元组

### 5. 区间摘要（Zone Maps）为什么有效

例如某页里 `age` 的值都在 `[60, 80]` 之间：

- 如果查询条件是 `age < 30`
- 那这一整页可以直接跳过

这类方法本质是在页级别做快速剪枝（pruning）。

### 6. 索引扫描（Index Scan）

#### 基本思想

- 通过索引定位满足条件的记录
- 再访问对应的数据页或记录

#### 什么时候有效

- 条件选择性高
- 返回记录不多
- 索引键和过滤条件匹配较好

#### 风险

- 如果命中记录太多，可能变成大量随机 I/O
- 这时可能还不如顺序扫描

### 7. 多索引扫描 / 位图扫描（Multi-Index Scan / Bitmap Scan）

当一个查询有多个谓词（predicates）时，可以：

1. 分别通过多个索引取出 record id 集合
2. 按条件逻辑做交集（intersect）或并集（union）
3. 再回表读取记录
4. 最后检查剩余谓词

典型例子：

- 用 `age` 索引找出 `age < 30`
- 用 `dept` 索引找出 `dept = 'CS'`
- 取交集
- 回表后再检查 `country = 'US'`

### 8. 选择算法的总体思路

课件列了从 A1 到 A10 的多种选择算法，本质上是在回答：

- 没索引还是有索引
- 聚簇索引（clustering index）还是二级索引（secondary index）
- 等值条件（equality）还是范围条件（comparison）
- 合取（conjunction）还是析取（disjunction）

记忆重点不是编号，而是下面这条原则：

**选择操作的核心是在“扫描代价”和“索引定位代价”之间做权衡。**

## 六、排序（Sorting）

### 1. 为什么数据库里经常需要排序

排序（sorting）不仅是为了 `ORDER BY`，还常用于：

- 去重（`DISTINCT`）
- 分组聚合（`GROUP BY`）
- 给 B+ 树批量装载（bulk loading）
- 某些连接算法，例如排序归并连接（sort-merge join）

### 2. 内存够时：内部排序

如果数据能全部放进内存，就直接用普通排序算法：

- 快速排序（quick-sort）等

这时问题相对简单，主要成本在 CPU。

### 3. 内存不够时：外部归并排序（External Merge Sort）

这是数据库排序的重点。

#### 阶段 1：生成初始有序段（Run Generation）

- 每次读入一块能放入内存的数据
- 在内存里排序
- 写回磁盘
- 形成多个有序段（sorted runs）

#### 阶段 2：归并（Merge）

- 把多个已排序 run 逐步归并成更大的 run
- 最终合成一个完整有序文件

### 4. run 是什么

run 可以理解为“已经局部排好序的一段数据”。

课件还强调了 run 中的内容有两种组织方式：

#### 早物化（Early Materialization）

- run 里直接放完整元组（tuple）

#### 晚物化（Late Materialization）

- run 里只放排序键（key）和记录标识（record id）
- 后面需要真正数据时再去取

### 5. 两路外部归并排序（2-Way External Merge Sort）

#### 核心特征

- 每次只归并 2 个 run
- 实现直观，但 pass 数较多

#### 成本特征

- 每一趟（pass）都要把所有页面读一遍、写一遍
- 所以每趟 I/O 成本大致是 `2N`

### 6. 一般的外部归并排序公式

设：

- 文件有 `N` 页
- 内存有 `B` 个缓冲页（buffer pages）

则：

- Pass 0 可生成大小约为 `B` 页的初始 run
- 后续每一趟最多做 `B-1` 路归并（K-way merge）

常见记法：

- 归并趟数约为 `⌈log_(B-1)(N / B)⌉`
- 总 pass 数约为 `1 + ⌈log_(B-1)(N / B)⌉`
- 总 I/O 成本约为 `2N × [1 + ⌈log_(B-1)(N / B)⌉]`

### 7. 课件例子：108 页、5 个缓冲页

设 `N = 108`，`B = 5`：

- Pass 0：生成每段 5 页的 run，最后一段 3 页
- Pass 1：每次 4 路归并，得到 20 页左右的 run，最后一段 8 页
- Pass 2：再 4 路归并，得到 80 页和 28 页两个 run
- Pass 3：最终合并成 108 页的有序文件

所以一共需要 4 个 pass（含 Pass 0）。

## 七、连接（Join）算法

这一部分是本讲重点中的重点。

### 1. 这讲关注什么样的连接

课件明确说，主要讨论：

- 二元连接（binary join）
- 内连接等值连接（inner equijoin）

很多其他连接都可以在这些算法上做变形。

### 2. 连接输出（Operator Output）有两种思路

#### 早物化（Early Materialization）

- 连接成功后，直接把左右元组需要的属性复制到新元组里
- 好处：上层算子不需要回到底表再取数据
- 代价：复制更多数据，占更多空间

#### 晚物化（Late Materialization）

- 只保留连接键（join key）和记录标识（RID, Record ID）
- 需要真正字段时再去取
- 对列存（column store）尤其友好

### 3. 连接代价分析的默认假设

设：

- `R` 有 `M` 页、`m` 条元组
- `S` 有 `N` 页、`n` 条元组

通常以 **I/O 次数** 作为主要成本指标。  
并且这讲里通常**忽略输出结果写出的代价**，因为输出大小强依赖数据本身，事先不容易精确估计。

### 4. 为什么通常让较小表做外表（Outer Table）

连接算法中经常区分：

- 外表（outer table）
- 内表（inner table）

经验原则是：

- 尽量让更小的表做外表
- “更小”优先看页数（pages），不是元组数（tuples）

原因是许多算法都要反复扫描内表，外表越小越省 I/O。

## 八、嵌套循环连接（Nested Loop Join）

### 1. 简单嵌套循环连接（Simple Nested Loop Join）

#### 基本思想

对外表 `R` 的每一条元组 `r`：

- 扫描内表 `S` 的所有元组 `s`
- 如果 `r` 和 `s` 满足连接条件，就输出

#### 代价

课件总结为：

- I/O 成本约为 `M + (m × N)`

含义是：

- 先读一遍 `R`
- `R` 的每个元组都可能导致把 `S` 扫一遍

#### 为什么它很差

- 外表每来一条元组，内表都要重新做顺序扫描
- 当外表元组很多时，代价爆炸

#### 例子

如果：

- `R`: `M = 1000`, `m = 100000`
- `S`: `N = 500`, `n = 40000`

课件估算其耗时大约是按小时计，显然非常慢。

### 2. 块嵌套循环连接（Block Nested Loop Join）

#### 基本思想

不是“按元组”处理外表，而是“按块（block/page）”处理外表：

- 取外表一个块
- 再扫描内表全部块

#### 代价

无额外缓冲优化时，常写成：

- `M + (M × N)`

比简单嵌套循环连接好很多，因为：

- 现在是“外表每页触发一次内表扫描”
- 而不是“外表每条元组触发一次内表扫描”

#### 课件例子

在同样例子下，课件估算大约为 50 秒，已经比简单嵌套循环连接好得多。

### 3. 带缓冲区的块嵌套循环连接

如果有 `B` 个缓冲页：

- 用 `B-2` 页装外表块
- 1 页装内表当前块
- 1 页做输出缓冲

此时代价常写成：

- `M + ⌈M / (B - 2)⌉ × N`

#### 特殊情况

如果外表能一次完整放进内存，即：

- `B > M + 2`

那么代价接近：

- `M + N`

因为：

- 外表读一遍放入内存
- 内表只需再扫一遍

这就是为什么**尽量缓存外表**会非常有效。

### 4. 索引嵌套循环连接（Index Nested Loop Join）

#### 基本思想

对外表每条元组，不再顺序扫内表，而是：

- 通过内表索引直接 probe（探测）匹配项

#### 代价

课件写法是：

- `M + (m × C)`

其中：

- `C` 是每次索引探测（index probe）的平均成本

#### 什么时候好用

- 内表连接键上已经有索引
- 外表不大
- 连接选择性较高

#### 记忆点

嵌套循环连接并不一定差，**差的是“没有索引还硬扫内表”**。

## 九、排序归并连接（Sort-Merge Join）

### 1. 两阶段

#### 阶段 1：排序（Sort）

- 把两张表都按连接键（join key）排序

#### 阶段 2：归并（Merge）

- 用两个游标（cursor）同步向前扫描
- 键值匹配时输出结果

### 2. 优点

- 如果两边已经按连接键有序，代价会很好
- 输出天然按连接键有序
- 对某些需要顺序结果的后续操作很友好

### 3. 风险和缺点

- 排序本身可能很贵
- 最坏情况下，如果大量元组连接键相同，回退和重复匹配会增加处理复杂度

### 4. 代价构成

总成本大致是：

- 排序 `R` 的成本
- 排序 `S` 的成本
- 再加一次归并扫描成本 `M + N`

即：

- `SortCost(R) + SortCost(S) + (M + N)`

### 5. 课件例子

课件中：

- `R: M = 1000`
- `S: N = 500`
- `B = 100`

两张表都能在 2-pass 排序内完成，最后总成本大约：

- `7500 IOs`

按课件假设折算，大约 `0.75 秒`。

### 6. 什么时候适合用

- 一张或两张表已经有序
- 可以利用索引顺序扫描拿到有序数据
- 输出本来就必须按连接键有序

## 十、哈希连接（Hash Join）

### 1. 基本哈希连接（Basic Hash Join）

#### 阶段 1：Build

- 扫描外表 `R`
- 按连接属性建立哈希表（hash table）

#### 阶段 2：Probe

- 扫描内表 `S`
- 对每条 `S` 中元组计算哈希值
- 到哈希表里找匹配项

### 2. 哈希表里存什么

课件给了两种典型设计：

#### 方案 1：完整元组（Full Tuple）

- 命中时不需要再回去取外表内容
- 但占更多内存

#### 方案 2：元组标识符（Tuple Identifier）

- 占更少空间
- 对列存更友好
- 当连接选择率低（join selectivity low）时也常更划算

### 3. Probe 阶段优化：布隆过滤器（Bloom Filter）

做法是：

- 在 build 阶段顺手构建 Bloom Filter
- probe 前先查 Bloom Filter
- 如果过滤器说“不可能存在”，就不必探测哈希表

优点：

- Bloom Filter 通常更小，更容易放进 CPU cache
- 可以减少大量无效 probe

课件还提到一个名字：

- 侧向信息传递（sideways information passing）

### 4. 基本哈希连接的适用前提

如果构建出的哈希表能放进内存，那么这种方法通常非常高效。

问题在于：

- 若表太大，放不下，就需要分区（partition）

### 5. Grace 哈希连接（Grace Hash Join）

这是**大表放不进内存时**的哈希连接方案。

#### 阶段 1：分区（Partitioning）

- 用同一个哈希函数，把 `R` 和 `S` 按连接键分别切成多个桶（buckets）

#### 阶段 2：逐桶连接（Per-Partition Join）

- 只把 `R` 的第 `i` 个桶和 `S` 的第 `i` 个桶拿来比较

这样做的关键逻辑是：

- 如果两个元组连接键相等
- 它们在同一个哈希函数下必然会进入对应的同号桶

所以不必跨桶比较，显著降低工作量。

### 6. 递归分区（Recursive Partitioning）

如果某个桶仍然太大，放不进内存：

- 再对这个桶用第二个哈希函数 `h2`
- 继续拆成更小分区

直到每个分区足以在内存中完成 build/probe。

### 7. 哈希连接成本

在缓冲区足够、分布比较均匀的理想情况下，课件给出的 Grace Hash Join 典型成本是：

- `3(M + N)`

含义是：

1. 分区阶段：读写两张表一次
2. probe 阶段：再读两张表一次

即总共大致三倍于两表页数和。

### 8. 课件例子

若：

- `M = 1000`
- `N = 500`

则成本约为：

- `3 × (1000 + 500) = 4500 IOs`

折算大约 `0.45 秒`，在课件比较中优于排序归并连接。

### 9. 哈希连接为什么常常很强

- 不必让数据先全局排序
- 等值连接（equijoin）非常适合哈希定位
- 若内存足够，通常比嵌套循环连接高效得多

但要注意：

- 它主要适合等值连接
- 如果哈希分布不均匀，会有数据倾斜（data skew）问题

## 十一、几种连接算法怎么选

### 1. 简单嵌套循环连接

- 几乎只适合教学或极小数据
- 真实系统中通常不是首选

### 2. 块嵌套循环连接

- 通用
- 不依赖索引
- 有足够缓冲区时改善明显

### 3. 索引嵌套循环连接

- 内表有好索引时很强
- 更适合一边较小、一边可快速 probe 的场景

### 4. 排序归并连接

- 输入已排序或输出需要排序时很有价值
- 也适合顺序访问比较友好的场景

### 5. 哈希连接

- 等值连接里非常常用
- 大规模分析场景很强
- 但依赖内存和哈希分布质量

## 十二、其他操作（Other Operations）

### 1. 去重（Duplicate Elimination）

去重通常有两大类做法：

#### 基于排序（Sorting）

- 排序后，相同值会相邻
- 只保留一份即可

#### 基于哈希（Hashing）

- 相同值会进入同一桶
- 再在桶内去重

课件还提到一个优化点：

- 在外部排序的 run 生成阶段和中间 merge 阶段，就可以提前删一部分重复值

### 2. 投影（Projection）

投影常和去重组合出现：

1. 先取出需要的列
2. 再做 duplicate elimination

### 3. 聚合（Aggregation）

聚合可以用：

- 排序
- 哈希

#### 一个重要优化：部分聚合（Partial Aggregation）

例如：

- `COUNT`、`SUM`、`MIN`、`MAX` 可以边扫描边维护局部结果
- `AVG` 可以维护 `sum + count`，最后再相除

### 4. 集合运算（Set Operations）

像：

- 并（union）
- 交（intersection）
- 差（difference）

通常可以借助：

- 排序后的 merge 风格算法
- 哈希 join 的变体

### 5. 关键字查询（Keyword Queries）

课件提到这类查询通常依赖：

- 倒排索引（inverted index）

### 6. 外连接（Outer Join）

外连接可以理解为：

- 先做 join
- 再补上未匹配的一侧元组
- 缺失属性用 `NULL` 填充（null-padded）

## 十三、考试和复习最该抓住的点

### 1. 一条 SQL 的执行主线

- Parser → Binder → Rewriter → Optimizer → Execution Engine → Storage Manager

### 2. 三种处理模型必须能比较

- 迭代器模型（Iterator Model）：一次一条，通用但调用开销大
- 物化模型（Materialization Model）：一次全部输出，适合小结果 OLTP
- 向量化模型（Vectorized Model）：一次一批，适合现代 OLAP

### 3. 访问方法必须能区分

- 顺序扫描（Sequential Scan）
- 索引扫描（Index Scan）
- 多索引/位图扫描（Multi-Index / Bitmap Scan）

### 4. 外部归并排序必须掌握

- 为什么要分 run
- 为什么每趟都要读写全文件
- pass 数和 I/O 成本怎么估

### 5. 连接算法必须会横向比较

- Simple Nested Loop Join：最慢，代价近似 `M + mN`
- Block Nested Loop Join：更实用，代价近似 `M + MN`
- Index Nested Loop Join：有索引时代价近似 `M + mC`
- Sort-Merge Join：代价是排序成本加归并成本
- Hash Join：典型代价近似 `3(M + N)`

### 6. 选择连接算法时的经验原则

- 小表做 outer table
- 优先看页数，不是先看元组数
- 有索引不一定最好，要看 probe 成本
- 已排序输入非常利于 sort-merge join
- 等值连接通常很适合 hash join

## 十四、最后用一句话串起来

查询处理（Query Processing）本质上是在回答四个问题：

- 这条 SQL 要拆成哪些操作符？
- 这些操作符之间怎样传数据？
- 每个操作符具体用什么物理算法？
- 在当前内存、I/O、索引和并发条件下，哪种方案总代价最低？

真正强的地方不在“知道有哪些算法”，而在于**能把执行模型、算法特点和代价分析连成一条线**。
