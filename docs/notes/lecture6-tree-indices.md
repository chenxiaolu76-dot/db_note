## 一、索引基础（Basic Concepts）

### 1. 什么是表索引
- 表索引是数据表中某些属性（列）的“副本（replica）”，并按特定规则组织/排序。
- 目的是让查询更高效，避免每次都全表扫描。
- 索引和主表逻辑上保持同步，由 DBMS 自动维护一致性。

### 2. 查询执行中的角色分工
- SQL 优化器负责选择“最优索引/索引组合”来执行查询。
- 用户不直接控制每次查询走哪个索引（除特殊 hint 场景）。

### 3. 索引数量的核心权衡
- 索引建太少：读查询慢。
- 索引建太多：
  - 存储开销上升（每个索引都要占空间）。
  - 维护开销上升（INSERT/UPDATE/DELETE 需要同步更新多个索引）。

## 二、为什么索引快：Table Scan vs Index Scan

课件用 `instructor` 表的 `id='45565'` 演示了两种路径：

### 1. 全表扫描（Table Scan）
- 从数据页顺序读，逐条比较 `id`。
- I/O 量与表大小近似线性关系。

### 2. 通过索引定位（Index）
- 先在索引结构中快速定位目标键（如 `45565`）。
- 再通过指针/记录地址访问目标数据。
- 点查性能显著提升，尤其是大表。

## 三、相关概念与分类（Related Concepts）

### 1. 有序索引 vs 哈希索引
- Ordered Index（有序索引）：键有序，支持点查与范围查。
- Hash Index（哈希索引）：适合点查，不擅长范围查。

### 2. 性能评估维度
- 访问类型：Point Query（点查询）、Range Query（范围查询）。
- 代价指标：查询时间、插入时间、删除时间。
- 空间指标：索引额外空间开销。

### 3. 聚簇与非聚簇
- Clustering Index（聚簇索引）：数据记录物理顺序与索引键顺序接近。
- Nonclustering/Secondary Index（非聚簇/二级索引）：索引顺序与数据物理顺序不一致。

### 4. Primary Index vs Primary Key（容易混淆）
- Primary Key：逻辑约束，要求唯一且非空。
- Primary Index（很多教材语境下）：通常指“聚簇索引/主访问路径”，不是“主键约束”的同义词。

### 5. Dense vs Sparse
- Dense（稠密索引）：几乎每个搜索键值（或每条记录）都在索引里有条目。
- Sparse（稀疏索引）：只给部分键建立条目（常见于按块建立入口）。

## 四、多级索引与复合键索引

### 1. 多级索引（Multilevel Index）
- 当单层索引过大，可在索引上再建索引，形成多级结构。
- B+ 树本质就是高效的多级索引结构。

### 2. 多属性索引（Composite Search Key）
- 例如在 `instructor(name, id)` 上建复合索引。
- 按字典序（lexicographic order）比较：
  - `(John, 12121) < (John, 13514)`
  - `(John, 13514) < (Peter, 11223)`
- 可支持：
  - 只按前导列 `name` 查询。
  - 按 `(name, id)` 联合查询。
- 实务记忆：复合索引常满足“最左前缀”访问规律。

## 五、B-Tree 家族概览

- B-Tree 家族是“平衡树”索引结构。
- 课件给出的时间线：
  - B-Tree（1971）
  - B+Tree（1973）
  - B*Tree（1977）
  - Blink-Tree（1981）

## 六、B+ 树结构（重点）

### 1. 节点类型
- Root Node（根节点）
- Internal Node（内部节点）
- Leaf Node（叶子节点）
- Data Blocks（数据块，叶节点指向数据记录或数据页）

### 2. B+ 树定义性质
设参数 `n` = 一个内部节点最多可有 `n` 个指针（即最多 `n` 个孩子）：

- 所有根到叶路径长度相同（严格平衡）。
- 非叶节点满足：`键数 = 孩子数 - 1`。
- 根节点约束：
  - 若根是叶子（整棵树只有一个节点）：键数范围 `0 ~ (n-1)`。
  - 若根不是叶子：孩子数范围 `2 ~ n`，键数范围 `1 ~ (n-1)`。
- 内部节点（非根）约束：
  - 孩子数范围：`[ceil(n/2), n]`
  - 键数范围：`[ceil(n/2)-1, n-1]`
- 叶节点（非根）约束：
  - 键数范围：`[ceil((n-1)/2), n-1]`
  - 叶子不再有“孩子节点”（通常还会有叶间链指针用于范围查询，这不计入孩子数）。

课件例子 `n=6`：
- 叶节点键值数 `[3,5]`
- 内部节点孩子数 `[3,6]`
- 根节点孩子数：`0`（树只有根且为空）或 `[2,6]`

做题常用 `n=3`（你最近那道题）：
- 根为非叶：孩子数 `[2,3]`，键数 `[1,2]`
- 内部节点（非根）：孩子数 `[2,3]`，键数 `[1,2]`
- 叶节点（非根）：键数 `[1,2]`
- 若根同时是叶子：键数 `[0,2]`

### 3. 节点内部格式
- 键：`K1 < K2 < ... < K(n-1)`
- 指针：`P1, P2, ..., Pn`
- 非叶路由语义（典型）：
  - `P1` 指向 `< K1` 的子树
  - `Pi` 指向 `[K(i-1), Ki)` 的子树
  - `Pn` 指向 `>= K(n-1)` 的子树

## 七、B+ 树查询（find）

### 1. 基本流程
1. 从根节点开始。
2. 在当前内部节点中找最小的 `i`，使得 `v <= Ki`。
3. 根据比较结果走到对应孩子指针。
4. 到达叶节点后查找是否存在键 `v`。
5. 找到返回记录指针；否则返回空。

### 2. 复杂度
- 树高为 `h`，查找复杂度约 `O(h)`。
- 因扇出大（每层分叉很多），`h` 通常很小，近似 `O(log N)` 且常数小。

## 八、B+ 树插入

### 1. 总体思路
1. 先 `find(K)` 找到目标叶节点 `L`。
2. 若 `L` 未满，按序插入 `(K, P)`。
3. 若 `L` 已满，分裂叶节点并把“新右叶最小键”上推到父节点。
4. 若父节点也满，递归分裂，必要时创建新根。

### 2. `insert_in_leaf`
- 若 `K` 小于当前叶最小键，插到最前。
- 否则找到不大于 `K` 的最大键 `Ki`，插到其后。
- 本质：保持叶内有序。

### 3. 叶分裂关键点（课件伪代码对应）
- 用临时空间 `T` 容纳“旧叶 + 新键”，共 `n` 个键值对。
- 将前半复制回 `L`，后半放入新叶 `L'`。
- 维护叶间链指针（`L.Pn` 与 `L'.Pn`）。
- 取 `L'` 的最小键 `K'`，执行 `insert_in_parent(L, K', L')`。

### 4. `insert_in_parent`
- 若 `L` 原本是根：新建根 `R`，内容为 `(L, K', L')`。
- 否则把 `(K', L')` 插到父节点中 `L` 指针的后面。
- 父满则继续分裂父节点并向上递归。

### 5. 课件示例
- 插入 `Adams`
- 插入 `Lamport`
- 都用于展示“叶分裂 + 父节点可能级联分裂”。

## 九、B+ 树删除

### 1. 总体入口
- `delete(K, P)`：先找到包含 `(K,P)` 的叶节点 `L`，再执行 `delete_entry(L, K, P)`。

### 2. 删除后的三类情况
1. 删除后节点仍满足最小占用：结束。
2. 删除后下溢（too few values/pointers），但可向兄弟借一个：重分配（redistribute）。
3. 删除后下溢且借不了：与兄弟合并（merge），并在父节点删除分隔键，可能向上级联。

### 3. 根节点特殊处理
- 若根删除后只剩一个孩子，则该孩子提升为新根，树高减 1。

### 4. 借位与合并的区分
- 内部节点和叶节点规则略不同（分隔键是否下移/替换方式不同）。
- 左兄弟借/右兄弟借都要同步更新父节点中的分隔键。
- 课件分别给了：
  - 与兄弟合并（内部/叶）
  - 从左兄弟借（内部/叶）
  - 从右兄弟借（内部/叶）

### 5. 课件示例
- 删除 `Srinivasan`
- 删除 `Singh` 和 `Wu`
- 删除 `Gold`
- 目的：观察删除触发的借位、合并与层高变化。

## 十、B+ 树性能观察（Observations）

### 1. 逻辑相邻 vs 物理相邻
- 逻辑上相近的键，不一定在磁盘物理位置相邻。
- B+ 树保证的是“逻辑有序+高效导航”，不是物理连续。

### 2. 树高为什么低
- 扇出高，层数少。
- 在块大小 4KB、键长 12B、指针 8B 的配置下，单节点可容纳大量键/指针，树高通常很小。

### 3. 规模与复杂度
- 查找、插入、删除都可在对数时间内完成（`O(log N)`）。
- 插入顺序会影响平均节点占用率（occupancy）。

## 十一、非唯一搜索键（Non-Unique Search Keys）

课件给出两种常见设计：

### 方案 A：每个键挂一个桶/指针列表
- 做法：键 -> 一组 tuple pointers。
- 优点：键存储节省。
- 缺点：重复值很多时，删除某条记录可能代价高，代码复杂。

### 方案 B：把键“唯一化”
- 做法：在搜索键后拼接 `record-id` 变为唯一键。
- 优点：插入/删除逻辑简单，工业界常用。
- 缺点：键更长，存储开销增大。

## 十二、B+ 树文件组织与二级索引

### 1. 记录迁移问题
- 若记录位置发生移动，所有存“记录物理指针”的二级索引都要更新。
- 在 B+ 树文件组织里，节点分裂可能导致大量记录迁移，更新成本高。

### 2. 课件给出的优化思路
- 二级索引不直接存记录指针，改存“主文件组织 B+ 树的搜索键”。
- 若该搜索键不唯一，再加 `record-id`。
- 代价：查询时多一次定位跳转。
- 收益：主文件分裂时，二级索引维护更便宜。

## 十三、字符串索引（Indexing Strings）

### 1. 变长键问题
- 字符串长度不固定，导致节点扇出可变。
- 分裂标准不应只看“条目个数”，更应看“空间利用率”。

### 2. 前缀压缩（Prefix Compression）
- 内部节点可只存足够区分子树的前缀，而非完整字符串。
- 例子：`Silas` 与 `Silberschatz` 可用前缀 `Silb` 分隔。
- 叶节点也可通过共享公共前缀进行压缩，提升空间利用率。

## 十四、B-Tree vs B+Tree（考试高频）

### 1. 数据放置位置
- B-Tree：内部节点和叶节点都可存记录（或记录指针）。
- B+Tree：记录（或记录指针）只在叶节点；内部节点只做导航。

### 2. 范围查询
- B+Tree 通常更适合范围查询（叶节点顺序访问友好）。

### 3. 扇出与树高
- B+Tree 内部节点不存记录，通常可容纳更多键，扇出更高，树更矮。

### 4. 工程实践
- 数据库系统中 B+Tree 使用更广泛。

## 十五、复习清单（建议背会）

1. 索引不是“越多越好”，要平衡读性能与写维护成本。  
2. 点查/范围查选择：有序索引 vs 哈希索引。  
3. 区分 `Primary Key`（约束）与 `Primary Index`（访问路径/聚簇语境）。  
4. B+ 树三个核心不变量：平衡、高扇出、最小占用率。  
5. 插入看“分裂+上推”，删除看“借位/合并+可能级联”。  
6. 重复键处理：桶列表 vs 键唯一化（+record-id）。  
7. 字符串索引要点：变长键、空间驱动分裂、前缀压缩。
